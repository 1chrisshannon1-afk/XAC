# Root Files Manifest

Every file at repo root exists because a tool, framework, or runtime convention requires it there. This document explains **why** each file must live at root and where its canonical source is.

XAC onboarding installs or generates these files. Projects should not create them manually.

---

## Category 1: XAC-Managed Tooling (synced from canonical source)

These files are **owned by XAC**. Their canonical source lives in `_Config_Project/tooling/` (project-specific values) or `_XAC_Base/ci/templates/` (shared templates). Root copies exist because the tools discover config by looking at the working directory / repo root.

| Root file | Why it must be at root | Canonical source | Sync mechanism |
|---|---|---|---|
| `pytest.ini` | pytest discovers config from `cwd` or `rootdir` | `_Config_Project/tooling/pytest.ini` | Manual sync or onboarding |
| `.pre-commit-config.yaml` | pre-commit framework looks for this at repo root | `_Config_Project/tooling/.pre-commit-config.yaml` | Manual sync or onboarding |
| `mypy.ini` | mypy loads config from `cwd` unless `--config-file` is passed | `_Config_Project/tooling/mypy.ini` | Manual sync or onboarding |
| `pyproject.toml` | pip, setuptools, coverage, and build tools expect this at root | `_Config_Project/tooling/pyproject.toml` | Manual sync or onboarding |
| `ruff.toml` | ruff discovers config by walking up from target files | `_Config_Project/tooling/ruff.toml` | Manual sync or onboarding |
| `.cursorrules` | Cursor IDE loads agent rules from repo root | Assembled from `_XAC_Base/cursor/cursorrules.base` + `_XAC_Config/cursor/cursorrules.project` | Assembled during onboarding |
| `.cursorignore` | Cursor IDE loads ignore patterns from repo root | Assembled from `_XAC_Base/cursor/cursorignore.base` + `_XAC_Config/cursor/cursorignore.project` | Assembled during onboarding |
| `.ci/config.ps1` | Thin wrapper sourced by CI engine and deploy scripts | Sources `_Config_Project/ci/config.ps1` | Created by onboarding |

## Category 2: Standard Tooling (generic, onboarded once)

These files are generic across projects. XAC onboarding should install default versions. They rarely change after initial setup.

| Root file | Why it must be at root | Notes |
|---|---|---|
| `.editorconfig` | EditorConfig spec requires it at or above the project root | Generic formatting rules (indent, charset, EOL). Same across all XAC projects. |
| `.prettierrc` | Prettier discovers config by walking up from target files | Generic formatting preferences. |
| `.prettierignore` | Prettier looks for ignore file at repo root | Excludes `_XAC_Base`, `_Config_Project`, `node_modules`, etc. |
| `.gitattributes` | Git reads this from repo root for EOL and diff behavior | Ensures `*.py` uses LF, `*.ps1` uses CRLF. Same across all XAC projects. |
| `.secrets.baseline` | detect-secrets stores its baseline at repo root | Generated by `detect-secrets scan`; XAC pre-commit hooks reference it. |
| `.dockerignore` | Docker reads this from build context root | Excludes dev artifacts from Docker build context. Project-specific paths. |
| `.gcloudignore` | gcloud CLI reads this from repo root during Cloud Build upload | Excludes dev artifacts from Cloud Build. Project-specific paths. |
| `eslint.config.mjs` | ESLint flat config is discovered from repo root | TypeScript/React linting rules. Generic for frontend projects. |
| `.python-version` | pyenv and version managers look for this at repo root | Must match `$CI_PYTHON_VERSION` in `_Config_Project/ci/config.ps1`. |
| `.nvmrc` | nvm looks for this at repo root | Must match `$CI_NODE_VERSION` in `_Config_Project/ci/config.ps1`. |
| `.gitignore` | Git reads this from repo root | Combines generic patterns (venv, __pycache__) with project-specific paths and credential patterns. |

## Category 3: Firebase / GCP (project-specific, required at root)

These files are dictated by Google tooling conventions. They contain project-specific values.

| Root file | Why it must be at root | Notes |
|---|---|---|
| `firebase.json` | Firebase CLI expects this at project root | References Firestore rules/indexes paths. Project-specific service IDs. |
| `firestore.rules` | Referenced by `firebase.json`; Firebase CLI resolves relative to root | Security rules for Firestore. |
| `.firebaserc` | Firebase CLI stores project aliases here | Contains `"default": "<gcp-project-id>"`. |

## Category 4: Application entry points (project-specific)

These are not infrastructure — they're the actual application. XAC does not manage these.

| Root file | Purpose |
|---|---|
| `entrypoint_backend.py` | Python application entry point |
| `conftest.py` | Root pytest conftest — sets `collect_ignore` for XAC folders, safe test env defaults |
| `requirements.txt` | Python dependencies |
| `package.json` / `package-lock.json` | Node.js dependencies and root-level lint/format scripts |
| `README.md` | Project documentation |
| `LICENSE` | Project license |

## Category 5: Local secrets (never committed)

These files are gitignored. They must never be committed to the repository. They are created locally by the developer.

| Root file | Purpose | How to create |
|---|---|---|
| `.env` | Local environment variables | Copy from `.env.example` |
| `.env.alloy` | Grafana Alloy credentials | Copy from `.env.alloy.example` |
| `local-dev-key.json` | GCP service account key for local development | `gcloud iam service-accounts keys create local-dev-key.json --iam-account=...` or use ADC |

---

## Key Principle

**No root-level stub scripts.** Docker, CI, deploy, and server scripts live in their canonical locations (`_Config_Project/docker/`, `_Config_Project/scripts/`, `_XAC_Config/ci/`). Users invoke them directly with their full path. This eliminates indirection and makes it obvious where logic lives.

## Adding Root Files to a New Project

XAC onboarding (`_XAC_Base/scripts/onboard-xac.ps1`) should install all Category 1 and Category 2 files. Category 3 files are created when the developer sets up Firebase. Category 4 files are created by the developer. Category 5 files are created locally and never committed.
